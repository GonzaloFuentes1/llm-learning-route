Reward Models: entrenamiento y uso

Un Reward Model (RM) es un LLM ajustado para puntuar respuestas en una escala escalar. Base del pipeline RLHF clásico y herramienta clave para filtrar datos sintéticos. Cubre entrenamiento con Bradley-Terry, scoring de respuestas y best-of-N.

1. El modelo Bradley-Terry

Un RM convierte comparaciones pareadas (A es mejor que B) en scores escalares. El modelo Bradley-Terry asume que la probabilidad de preferir A sobre B es:

P(A > B) = σ(r(A) − r(B))

donde r(x) es el score del RM para la respuesta x y σ es la función sigmoide. El entrenamiento minimiza la cross-entropy sobre pares (chosen, rejected).

Tipo de RMArquitecturaCuándo usarlo
Scalar RMLLM + linear head → 1 númeroRLHF/PPO, filtrado de datos, best-of-N
Sequence ClassifierLLM + clasificación binariaMás simple, menos expresivo
LLM-as-JudgePrompt a un LLM grandeMás caro, bueno para tareas abiertas

2. Formato de datos

El RM necesita pares (chosen, rejected) con el mismo prompt. El modelo aprende a asignar mayor score al chosen.

json — formato preference para RM
{
  "prompt":   "¿Cómo funciona el sistema de salud en Chile?",
  "chosen":   "El sistema de salud chileno es mixto: el sector público (Fonasa) cubre al 80% de la población con financiamiento estatal, mientras el sector privado (Isapres) atiende al 20% restante. Para acceder a Fonasa necesitas...",
  "rejected": "Chile tiene hospitales y clínicas. Hay Fonasa para los pobres e Isapres para los ricos."
}
python — cargar datos para RewardTrainer
from datasets import load_dataset
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-3B-Instruct")

def tokenize_pair(example):
    """RewardTrainer espera input_ids_chosen / input_ids_rejected."""
    chosen_msgs  = [{"role": "user", "content": example["prompt"]},
                    {"role": "assistant", "content": example["chosen"]}]
    rejected_msgs = [{"role": "user", "content": example["prompt"]},
                     {"role": "assistant", "content": example["rejected"]}]

    chosen  = tokenizer.apply_chat_template(chosen_msgs,  tokenize=True, truncation=True, max_length=2048)
    rejected = tokenizer.apply_chat_template(rejected_msgs, tokenize=True, truncation=True, max_length=2048)

    return {
        "input_ids_chosen":   chosen,
        "attention_mask_chosen":  [1] * len(chosen),
        "input_ids_rejected": rejected,
        "attention_mask_rejected": [1] * len(rejected),
    }

dataset = load_dataset("json", data_files="data/preference_pairs.jsonl", split="train")
dataset = dataset.map(tokenize_pair, remove_columns=dataset.column_names)

3. Entrenamiento del RM

python — train_rm.py
import torch
from transformers import AutoModelForSequenceClassification, AutoTokenizer
from trl import RewardTrainer, RewardConfig
from peft import LoraConfig

BASE_MODEL = "meta-llama/Llama-3.2-3B-Instruct"
OUTPUT_DIR = "models/reward-model-es"

# AutoModelForSequenceClassification con num_labels=1 → scalar output
model = AutoModelForSequenceClassification.from_pretrained(
    BASE_MODEL,
    num_labels=1,           # score escalar
    torch_dtype=torch.bfloat16,
    device_map="auto",
)
model.config.pad_token_id = model.config.eos_token_id

tokenizer = AutoTokenizer.from_pretrained(BASE_MODEL)
tokenizer.pad_token = tokenizer.eos_token

lora_config = LoraConfig(
    r=32,
    lora_alpha=64,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    bias="none",
)

training_args = RewardConfig(
    output_dir=OUTPUT_DIR,
    num_train_epochs=1,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=1e-5,
    bf16=True,
    max_length=2048,
    remove_unused_columns=False,
    logging_steps=20,
    save_steps=200,
    report_to="wandb",
)

trainer = RewardTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,    # dataset ya tokenizado arriba
    tokenizer=tokenizer,
    peft_config=lora_config,
)

trainer.train()
trainer.save_model(OUTPUT_DIR)
¿Cuántos datos necesito? Un RM útil para filtrado básico puede entrenarse con 10K–50K pares de alta calidad. Para RLHF/PPO estable necesitas 100K+. Empieza con UltraFeedback (64K pares) y añade datos propios del dominio.

4. Usar el RM para puntuar respuestas

python — score_responses.py
import torch
from transformers import AutoModelForSequenceClassification, AutoTokenizer

RM_PATH = "models/reward-model-es"

tokenizer = AutoTokenizer.from_pretrained(RM_PATH)
rm = AutoModelForSequenceClassification.from_pretrained(
    RM_PATH,
    num_labels=1,
    torch_dtype=torch.bfloat16,
    device_map="cuda",
)
rm.eval()


def score(prompt: str, response: str) -> float:
    """Devuelve el score del RM para un par (prompt, response)."""
    messages = [
        {"role": "user",      "content": prompt},
        {"role": "assistant", "content": response},
    ]
    tokens = tokenizer.apply_chat_template(
        messages, tokenize=True, return_tensors="pt", max_length=2048, truncation=True
    ).to(rm.device)

    with torch.no_grad():
        output = rm(input_ids=tokens)
    return output.logits[0, 0].item()


def best_of_n(prompt: str, responses: list[str], n: int = None) -> tuple[str, float]:
    """Selecciona la mejor respuesta según el RM."""
    scored = [(r, score(prompt, r)) for r in responses]
    scored.sort(key=lambda x: x[1], reverse=True)
    return scored[0]   # (mejor respuesta, score)


# Ejemplo
prompt = "Explica la reforma previsional en Chile"
responses = [
    "La reforma busca aumentar la tasa de cotización del 10% al 16%...",
    "No sé exactamente pero creo que cambian las AFP",
    "En 2024, el Congreso chileno debatió una reforma que modifica el sistema de AFP...",
]

best, best_score = best_of_n(prompt, responses)
print(f"Mejor respuesta (score={best_score:.3f}):\n{best}")

5. Generación sintética con RM (best-of-N + delta DPO)

Técnica usada en Tülu 3 / OLMo 3: generar N respuestas por prompt, puntuar con RM, y construir pares DPO filtrando por diferencia de score (δ).

python — online_dpo_data.py
import json
from openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1", api_key="token")  # modelo local con vLLM

DELTA = 1.5   # diferencia mínima de score para incluir el par
N     = 8     # respuestas generadas por prompt


def generate_n_responses(prompt: str, n: int) -> list[str]:
    responses = []
    for _ in range(n):
        resp = client.chat.completions.create(
            model="models/sft-checkpoint",
            messages=[{"role": "user", "content": prompt}],
            temperature=0.9,
            max_tokens=1024,
        )
        responses.append(resp.choices[0].message.content)
    return responses


def build_dpo_pairs(prompts: list[str], delta: float = DELTA) -> list[dict]:
    """Para cada prompt: genera N respuestas, puntúa, filtra por delta y construye pares DPO."""
    pairs = []
    for prompt in prompts:
        responses = generate_n_responses(prompt, N)
        scored = [(r, score(prompt, r)) for r in responses]   # score() del bloque anterior
        scored.sort(key=lambda x: x[1], reverse=True)

        best_resp,  best_score  = scored[0]
        worst_resp, worst_score = scored[-1]

        # Solo incluir si la diferencia supera el umbral delta
        if best_score - worst_score >= delta:
            pairs.append({
                "prompt":   [{"role": "user", "content": prompt}],
                "chosen":   [{"role": "assistant", "content": best_resp}],
                "rejected": [{"role": "assistant", "content": worst_resp}],
                "score_chosen":   best_score,
                "score_rejected": worst_score,
            })

    return pairs


# Cargar prompts y generar dataset
prompts = [line.strip() for line in open("data/prompts_es.txt")]
dpo_data = build_dpo_pairs(prompts)

with open("data/online_dpo_delta_filtered.jsonl", "w") as f:
    for pair in dpo_data:
        f.write(json.dumps(pair, ensure_ascii=False) + "\n")

print(f"Pares DPO generados: {len(dpo_data)} / {len(prompts)} prompts ({100*len(dpo_data)/len(prompts):.0f}%)")
El truco del delta (δ): si el RM puntúa casi igual el chosen y el rejected, el par no aporta señal útil al DPO. Filtrar por δ ≥ 1.0–2.0 mejora la calidad del dataset a costa de reducir cantidad. OLMo 3 / Tülu 3 usan este pipeline en múltiples iteraciones (online DPO).

6. Reward hacking y ley de Goodhart

"When a measure becomes a target, it ceases to be a good measure." El modelo aprende a explotar debilidades del RM en lugar de ser genuinamente mejor.

SíntomaCausaMitigación
Respuestas largas pero vacíasRM favorece longitudAñadir penalización por longitud al RM
Listas de bullets en cualquier respuestaRM favorece formato estructuradoDiversificar ejemplos de entrenamiento del RM
Scores del RM suben pero benchmark bajaRM está saturado / fuera de distribuciónReentrenar RM con datos frescos; usar KL penalty alto
Respuestas demasiado cortas (KTO)Modelo aprende a minimizar el rejectedAumentar β en DPO
