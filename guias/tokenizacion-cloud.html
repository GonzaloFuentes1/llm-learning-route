Tokenización en Cloud

Convertir un corpus de texto crudo en archivos de token IDs listos para entrenamiento, a escala de cientos de miles de millones de tokens. Usando Datatrove (HuggingFace) con paralelismo en cluster.

1. El problema de escala

Un corpus típico para pretraining tiene entre 100B y 15T tokens. Tokenizar secuencialmente con un solo proceso tardaría días. La solución es distribuir el trabajo: cada worker tokeniza un subconjunto de archivos en paralelo.

FineWeb (2024) tokeniznó ~15T tokens usando Datatrove con cientos de workers en paralelo. El resultado son archivos .ds (DatasetDocument Sequence), memory-mapped, que se pueden leer directamente durante el entrenamiento sin cargar todo a RAM.

Flujo típico: texto crudo (JSONL) → DocumentTokenizer → archivos .ds → DataLoader durante training.

3. Pipeline de tokenización con Datatrove

Datatrove usa un modelo de pipeline: defines una lista de pasos y un executor que los corre en paralelo sobre tus datos.

python — tokenize.py
from datatrove.executor import LocalPipelineExecutor, SlurmPipelineExecutor
from datatrove.pipeline.readers import JsonlReader
from datatrove.pipeline.tokens import DocumentTokenizer


def get_pipeline(input_folder: str, output_folder: str, tokenizer_name: str):
    return [
        JsonlReader(
            data_folder=input_folder,
            text_key="text",           # campo de texto en el JSONL
            id_key="id",
            glob_pattern="*.jsonl",
        ),
        DocumentTokenizer(
            output_folder=output_folder,
            tokenizer_name_or_path=tokenizer_name,
            eos_token="</s>",
            max_tokens_per_file=1_000_000_000,  # 1B tokens por archivo .ds
        ),
    ]


if __name__ == "__main__":
    executor = LocalPipelineExecutor(
        pipeline=get_pipeline(
            input_folder="data/raw/es/",
            output_folder="data/tokens/es/",
            tokenizer_name="meta-llama/Llama-2-7b-hf",
        ),
        tasks=64,    # número de shards/tareas
        workers=16,  # procesos en paralelo (ajustar a CPUs disponibles)
        logging_dir="logs/tokenization/",
    )
    executor.run()
tasks vs workers: tasks divide los datos en N partes; workers define cuántas corren en paralelo. Con tasks=64, workers=16, se procesan 16 shards simultáneamente hasta completar las 64.

4. Lanzar en SLURM con SlurmPipelineExecutor

Para escalar a cientos de nodos, reemplaza LocalPipelineExecutor por SlurmPipelineExecutor. Datatrove genera y envía los jobs automáticamente.

python — tokenize_slurm.py
from datatrove.executor import SlurmPipelineExecutor
from datatrove.pipeline.readers import JsonlReader
from datatrove.pipeline.tokens import DocumentTokenizer

executor = SlurmPipelineExecutor(
    pipeline=[
        JsonlReader(
            data_folder="/data/raw/",
            text_key="text",
            glob_pattern="**/*.jsonl",
        ),
        DocumentTokenizer(
            output_folder="/data/tokens/",
            tokenizer_name_or_path="meta-llama/Llama-2-7b-hf",
            eos_token="</s>",
            max_tokens_per_file=2_000_000_000,
        ),
    ],
    tasks=512,
    workers=64,           # workers por job
    time="04:00:00",      # tiempo límite por job SLURM
    partition="cpu",
    cpus_per_task=2,
    mem_per_cpu_gb=4,
    logging_dir="logs/slurm_tokenization/",
    job_name="tokenize_es",
)
executor.run()

5. Alternativa: tokenización paralela custom

Para mayor control o datasets pequeños (~10B tokens), puedes usar multiprocessing de Python directamente con el tokenizador de HuggingFace.

python — tokenize_simple.py
import json
import numpy as np
from pathlib import Path
from multiprocessing import Pool
from transformers import AutoTokenizer

TOKENIZER_PATH = "meta-llama/Llama-2-7b-hf"
INPUT_DIR = Path("data/raw/")
OUTPUT_DIR = Path("data/tokens/")
OUTPUT_DIR.mkdir(parents=True, exist_ok=True)

tokenizer = None  # se inicializa en cada worker

def init_worker():
    global tokenizer
    tokenizer = AutoTokenizer.from_pretrained(TOKENIZER_PATH)

def tokenize_file(jsonl_path: Path) -> dict:
    all_ids = []
    with open(jsonl_path) as f:
        for line in f:
            doc = json.loads(line)
            ids = tokenizer.encode(doc["text"], add_special_tokens=False)
            ids.append(tokenizer.eos_token_id)
            all_ids.extend(ids)

    out_path = OUTPUT_DIR / jsonl_path.with_suffix(".npy").name
    np.save(out_path, np.array(all_ids, dtype=np.uint16))
    return {"file": str(jsonl_path), "tokens": len(all_ids)}

if __name__ == "__main__":
    files = list(INPUT_DIR.glob("**/*.jsonl"))
    print(f"Tokenizando {len(files)} archivos...")

    with Pool(processes=16, initializer=init_worker) as pool:
        results = pool.map(tokenize_file, files)

    total = sum(r["tokens"] for r in results)
    print(f"Total: {total:,} tokens ({total/1e9:.2f}B)")
uint16 vs int32: Si tu vocabulario tiene ≤ 65535 tokens usa uint16 (2 bytes/token). Vocabularios más grandes (como los de 128k tokens de LLaMA-3) requieren int32 (4 bytes/token).

6. Verificar cobertura del vocabulario

La "fertilidad" del tokenizador mide cuántos tokens genera por palabra. Un tokenizador diseñado para inglés puede generar el doble de tokens para texto en español, lo que reduce la eficiencia y el contexto efectivo.

python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")

# Textos de prueba en es/pt
texts_es = [
    "La tokenización es fundamental para el rendimiento en español.",
    "Los modelos de lenguaje grande requieren datos de alta calidad.",
    "El procesamiento del lenguaje natural en latinoamérica es un reto.",
]

# Medir fertilidad: tokens / palabras
for text in texts_es:
    words = len(text.split())
    tokens = len(tokenizer.encode(text, add_special_tokens=False))
    fertility = tokens / words
    print(f"Fertilidad: {fertility:.2f} tokens/palabra  |  '{text[:40]}...'")

# Comparar contra tokenizador nativo es/pt
from transformers import AutoTokenizer
tok_es = AutoTokenizer.from_pretrained("dccuchile/bert-base-spanish-wwm-cased")
# Un tokenizador para español debería tener fertilidad ~1.2-1.5x
# El tokenizador inglés puede tener 2-3x para texto en español
Regla de oro: Para un LLM Latam de calidad, la fertilidad en español/portugués debería ser ≤ 1.5x la fertilidad en inglés para el mismo texto. Si es mayor, considera entrenar un tokenizador propio o usar uno multilingüe como el de mGPT o BLOOM.
