Código de Entrenamiento

Loop de pretraining con HuggingFace Accelerate y DeepSpeed ZeRO. Cubre configuración multi-GPU, gradient accumulation, precisión mixta bf16, checkpointing y cómo interpretar la loss curve.

1. Configurar Accelerate

Accelerate abstrae la complejidad de multi-GPU, DeepSpeed y FSDP. Una sola línea de config determina cómo se distribuye el entrenamiento.

bash
pip install accelerate deepspeed

# Generar config interactivamente (responde las preguntas)
accelerate config --config_file configs/deepspeed_zero2.yaml

# O crear el archivo directamente:
yaml — configs/deepspeed_zero2.yaml
compute_environment: LOCAL_MACHINE
distributed_type: DEEPSPEED
deepspeed_config:
  deepspeed_multinode_launcher: standard
  gradient_accumulation_steps: 4
  gradient_clipping: 1.0
  offload_optimizer_device: none
  offload_param_device: none
  zero3_init_flag: false
  zero_stage: 2
machine_rank: 0
main_training_function: main
mixed_precision: bf16
num_machines: 1
num_processes: 4          # = número de GPUs
use_cpu: false

2. Training loop básico

python — train.py
import math
import torch
from torch.utils.data import DataLoader
from transformers import AutoModelForCausalLM, AutoTokenizer, get_cosine_schedule_with_warmup
from accelerate import Accelerator
import wandb


def main():
    accelerator = Accelerator(
        gradient_accumulation_steps=4,
        mixed_precision="bf16",
        log_with="wandb",
    )

    # Inicializar W&B solo en el proceso principal
    if accelerator.is_main_process:
        wandb.init(project="latamgpt", name="pretraining-v1")

    # Cargar modelo y tokenizador
    model = AutoModelForCausalLM.from_pretrained(
        "meta-llama/Llama-2-7b-hf",
        torch_dtype=torch.bfloat16,
        attn_implementation="flash_attention_2",  # requiere flash-attn
    )

    tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")

    # Optimizer con weight decay selectivo (no aplicar a bias y layernorm)
    decay_params = [p for n, p in model.named_parameters() if "bias" not in n and "norm" not in n]
    no_decay_params = [p for n, p in model.named_parameters() if "bias" in n or "norm" in n]

    optimizer = torch.optim.AdamW(
        [{"params": decay_params, "weight_decay": 0.1},
         {"params": no_decay_params, "weight_decay": 0.0}],
        lr=3e-4,
        betas=(0.9, 0.95),
        eps=1e-8,
    )

    # Dataloader (tu dataset tokenizado)
    train_dataloader = DataLoader(dataset, batch_size=4, shuffle=True)

    # LR scheduler: warmup + cosine decay
    num_training_steps = len(train_dataloader) * NUM_EPOCHS
    num_warmup_steps = int(0.01 * num_training_steps)  # 1% de warmup
    scheduler = get_cosine_schedule_with_warmup(optimizer, num_warmup_steps, num_training_steps)

    # Preparar para distribución (Accelerate maneja DDP / DeepSpeed)
    model, optimizer, train_dataloader, scheduler = accelerator.prepare(
        model, optimizer, train_dataloader, scheduler
    )

    global_step = 0
    model.train()

    for epoch in range(NUM_EPOCHS):
        for step, batch in enumerate(train_dataloader):
            with accelerator.accumulate(model):  # gradient accumulation
                input_ids = batch["input_ids"]
                labels = batch["labels"]

                outputs = model(input_ids=input_ids, labels=labels)
                loss = outputs.loss

                accelerator.backward(loss)

                # Gradient clipping (solo aplica al paso real, no acumulación)
                if accelerator.sync_gradients:
                    grad_norm = accelerator.clip_grad_norm_(model.parameters(), max_norm=1.0)

                optimizer.step()
                scheduler.step()
                optimizer.zero_grad()

            # Logging
            if global_step % 10 == 0 and accelerator.is_main_process:
                wandb.log({
                    "train/loss": loss.item(),
                    "train/perplexity": math.exp(loss.item()),
                    "train/lr": scheduler.get_last_lr()[0],
                    "train/grad_norm": grad_norm.item() if accelerator.sync_gradients else None,
                    "train/step": global_step,
                })

            # Checkpoint cada 1000 pasos
            if global_step % 1000 == 0 and global_step > 0:
                if accelerator.is_main_process:
                    unwrapped = accelerator.unwrap_model(model)
                    unwrapped.save_pretrained(f"checkpoints/step_{global_step}")
                    tokenizer.save_pretrained(f"checkpoints/step_{global_step}")

            global_step += 1


if __name__ == "__main__":
    main()

5. Interpretar la loss curve

La forma de la loss curve durante pretraining es una señal diagnóstica crítica. Aquí lo que esperar:

text — fases típicas del training
Fase 1 — Warmup (primeros ~500 steps):
  Loss: baja rápidamente de ~12-14 a ~3-5
  LR: sube linealmente hasta el peak (ej: 3e-4)
  Acción: normal, no intervenir

Fase 2 — Descenso estable (bulk del training):
  Loss: baja gradualmente, ~0.001-0.01 por 1000 steps
  LR: decae con cosine schedule
  Gradient norm: estable, tipicamente 0.5-2.0
  Acción: monitorear, hacer checkpoints

Fase 3 — Loss spike (señal de alerta):
  Loss: sube bruscamente +0.3 o más en pocos steps
  Grad norm: puede subir a 10-100x lo normal
  Causas: batch con texto corrupto, LR muy alto, overflow en bf16
  Acción: ver debugging.html

Fase 4 — Cooldown (últimos ~5-10% del training):
  Loss: baja más rápido al reducir LR al mínimo
  Acción: no interrumpir, es cuando el modelo "consolida"
Perplexity como métrica: PPL = exp(loss). Un buen modelo de lenguaje en español alcanza PPL ≈ 8-15 en el set de validación. PPL > 50 sugiere problemas en los datos o configuración.

6. PyTorch Lightning — training loop sin boilerplate

Lightning organiza el código de entrenamiento en un LightningModule (modelo + lógica de training) y un Trainer (infraestructura: GPUs, precisión, checkpoints, logging). El mismo código corre en 1 GPU o 64 sin cambios.

bash
pip install lightning wandb
python — train_lightning.py
import math
import torch
import lightning as L
from transformers import AutoModelForCausalLM, get_cosine_schedule_with_warmup


class LLMPretrainer(L.LightningModule):
    def __init__(self, model_name: str, lr: float = 3e-4, warmup_steps: int = 1000):
        super().__init__()
        self.save_hyperparameters()
        self.model = AutoModelForCausalLM.from_pretrained(
            model_name,
            torch_dtype=torch.bfloat16,
            attn_implementation="flash_attention_2",
        )

    def training_step(self, batch, batch_idx):
        loss = self.model(**batch).loss
        ppl = math.exp(min(loss.item(), 20))
        self.log_dict({"train/loss": loss, "train/ppl": ppl}, on_step=True, prog_bar=True)
        return loss

    def validation_step(self, batch, batch_idx):
        loss = self.model(**batch).loss
        self.log("val/loss", loss, on_epoch=True, sync_dist=True)

    def configure_optimizers(self):
        # Weight decay solo sobre parámetros que no son bias ni layernorm
        decay = [p for n, p in self.named_parameters() if "bias" not in n and "norm" not in n]
        no_decay = [p for n, p in self.named_parameters() if "bias" in n or "norm" in n]
        optimizer = torch.optim.AdamW(
            [{"params": decay, "weight_decay": 0.1}, {"params": no_decay, "weight_decay": 0.0}],
            lr=self.hparams.lr, betas=(0.9, 0.95),
        )
        scheduler = get_cosine_schedule_with_warmup(
            optimizer,
            num_warmup_steps=self.hparams.warmup_steps,
            num_training_steps=self.trainer.estimated_stepping_batches,
        )
        return [optimizer], [{"scheduler": scheduler, "interval": "step"}]


# Configurar Trainer
trainer = L.Trainer(
    max_steps=100_000,
    precision="bf16-mixed",
    devices=4,                                  # GPUs
    strategy="deepspeed_stage_2",               # DeepSpeed ZeRO-2
    accumulate_grad_batches=8,
    gradient_clip_val=1.0,
    val_check_interval=1_000,
    log_every_n_steps=10,
    callbacks=[
        L.pytorch.callbacks.ModelCheckpoint(
            dirpath="checkpoints/",
            every_n_train_steps=1_000,
            save_top_k=3,
            monitor="val/loss",
        ),
        L.pytorch.callbacks.LearningRateMonitor(logging_interval="step"),
    ],
    logger=L.pytorch.loggers.WandbLogger(project="latamgpt", log_model=False),
)

model = LLMPretrainer("meta-llama/Llama-2-7b-hf")
trainer.fit(model, train_dataloader, val_dataloader)
Lightning vs Accelerate: Accelerate te da más control sobre el loop y es lo que usa la mayoría de repos industriales (OLMo, TRL). Lightning reduce boilerplate y facilita organizar experimentos. Para investigación y prototipos rápidos, Lightning es más ergonómico. Para producción y mayor flexibilidad, Accelerate.

7. Hydra — gestión de configuraciones

Hydra (Meta Research) compone configuraciones YAML jerárquicas y permite overrides desde la CLI. Elimina el `argparse` y los configs hardcodeados: cada experimento queda completamente reproducible con un solo comando.

bash
pip install hydra-core omegaconf
yaml — configs/train.yaml
defaults:
  - model: llama7b        # carga configs/model/llama7b.yaml
  - data: latam_web_v3    # carga configs/data/latam_web_v3.yaml
  - _self_

training:
  learning_rate: 3e-4
  batch_size: 4
  gradient_accumulation_steps: 8
  max_steps: 100_000
  warmup_steps: 1_000
  weight_decay: 0.1
  precision: bf16

hardware:
  num_gpus: 4
  strategy: deepspeed_stage_2

logging:
  project: latamgpt
  run_name: ${model.name}-${now:%Y%m%d_%H%M%S}
yaml — configs/model/llama7b.yaml
name: llama7b
hf_path: meta-llama/Llama-2-7b-hf
max_seq_len: 2048
attn_impl: flash_attention_2
python — train.py con Hydra
import hydra
from omegaconf import DictConfig, OmegaConf
import torch
from transformers import AutoModelForCausalLM

@hydra.main(config_path="configs", config_name="train", version_base=None)
def main(cfg: DictConfig) -> None:
    # El config se convierte en un dict anidado accesible con dot notation
    print(OmegaConf.to_yaml(cfg))   # ver config efectivo del run

    model = AutoModelForCausalLM.from_pretrained(
        cfg.model.hf_path,
        torch_dtype=torch.bfloat16,
    )
    optimizer = torch.optim.AdamW(
        model.parameters(),
        lr=cfg.training.learning_rate,
        weight_decay=cfg.training.weight_decay,
    )
    # ... resto del training loop

if __name__ == "__main__":
    main()
bash — overrides y sweeps desde CLI
# Lanzar con config base
python train.py

# Override puntual de cualquier valor
python train.py training.learning_rate=1e-4 hardware.num_gpus=8

# Cambiar modelo completo (carga configs/model/llama13b.yaml)
python train.py model=llama13b

# Multi-run: sweep de learning rates (lanza 3 jobs en serie)
python train.py -m training.learning_rate=1e-4,3e-4,1e-3

# El output de cada run se guarda en outputs/YYYY-MM-DD/HH-MM-SS/
# con el .hydra/config.yaml que registra exactamente qué parámetros se usaron
Reproducibilidad: Hydra guarda automáticamente el config completo en cada run. En 6 meses puedes reproducir cualquier experimento exactamente con el comando del log de W&B + el .hydra/config.yaml que quedó guardado.

8. torchtune — fine-tuning nativo de Meta

torchtune es la librería oficial de Meta para fine-tuning de LLMs. A diferencia de Lightning o Accelerate, está diseñada específicamente para fine-tuning (full, LoRA, QLoRA) con recetas (recipes) listas para usar sobre modelos como LLaMA, Mistral y Gemma. Usa Hydra internamente para los configs.

bash — instalar y listar recetas disponibles
pip install torchtune

# Ver todas las recetas disponibles
tune ls

# Recetas principales:
# full_finetune_single_device       → fine-tuning completo en 1 GPU
# full_finetune_distributed         → fine-tuning completo multi-GPU (FSDP2)
# lora_finetune_single_device       → LoRA en 1 GPU
# lora_finetune_distributed         → LoRA multi-GPU
# qlora_finetune_single_device      → QLoRA en 1 GPU (4-bit)
# generate                          → inferencia con modelo fine-tuneado
bash — LoRA fine-tuning con receta pre-construida
# Descargar config base para LoRA en LLaMA 3.2 3B
tune cp llama3_2/3B_lora_single_device ./configs/lora_llama3.yaml

# Editar el YAML con tu dataset y parámetros (vi o VS Code)
# Luego lanzar:
tune run lora_finetune_single_device --config configs/lora_llama3.yaml

# Multi-GPU con FSDP2 (4 GPUs):
tune run --nproc_per_node 4 lora_finetune_distributed --config configs/lora_llama3.yaml

torchtune vs TRL (HuggingFace): TRL (SFTTrainer, DPOTrainer) tiene más integraciones y es más flexible para post-training completo (SFT+DPO+RLHF). torchtune tiene recetas más limpias, mejor soporte de cuantización nativa con torchao, y es lo que Meta usa internamente para LLaMA.

9. ROCm — entrenamiento en GPUs AMD

ROCm (Radeon Open Compute) es la plataforma de cómputo de AMD, análoga a CUDA. PyTorch tiene soporte oficial via HIP: el código Python es idéntico a CUDA (tensor.cuda(), torch.device("cuda")), HIP traduce las llamadas al runtime AMD. Los modelos MI250X y MI300X son los targets principales para clusters de AMD.

bash — instalar PyTorch con ROCm
# Verificar versión de ROCm instalada en el cluster
rocminfo | grep "ROCm"
# o:

pip install torch torchvision torchaudio \
