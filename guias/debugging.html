Debugging de Entrenamiento

Diagnóstico y resolución de los problemas más comunes en pretraining y fine-tuning: loss spikes, gradient norms, NaN/Inf, OOM errors y memory profiling con torch.profiler.

1. Loggear las métricas correctas desde el inicio

El 80% de los problemas de training se detectan simplemente teniendo las métricas correctas en W&B. Agrega esto a tu loop antes de lanzar cualquier run largo:

python — métricas de diagnóstico
import torch
import wandb
import math


def log_training_step(
    accelerator, model, loss, optimizer, scheduler, global_step, log_every=10
):
    if global_step % log_every != 0:
        return

    # Gradient norm total
    total_norm = 0.0
    for p in model.parameters():
        if p.grad is not None:
            total_norm += p.grad.data.norm(2).item() ** 2
    total_norm = total_norm ** 0.5

    # Detectar NaN/Inf en gradientes
    has_nan_grad = any(
        p.grad is not None and torch.isnan(p.grad).any()
        for p in model.parameters()
    )
    has_inf_grad = any(
        p.grad is not None and torch.isinf(p.grad).any()
        for p in model.parameters()
    )

    if accelerator.is_main_process:
        wandb.log({
            "train/loss": loss.item(),
            "train/perplexity": min(math.exp(loss.item()), 1e6),
            "train/grad_norm": total_norm,
            "train/lr": scheduler.get_last_lr()[0],
            "debug/nan_grad": int(has_nan_grad),
            "debug/inf_grad": int(has_inf_grad),
        }, step=global_step)

        # Alerta en consola si hay problema
        if has_nan_grad or has_inf_grad:
            print(f"⚠️  Step {global_step}: {'NaN' if has_nan_grad else 'Inf'} en gradientes!")
        if total_norm > 10.0:
            print(f"⚠️  Step {global_step}: Gradient norm alta ({total_norm:.2f})")

2. Loss spikes

Una subida brusca de la loss (>0.3 en pocos steps) interrumpe el training. Las causas más comunes:

text — diagnóstico de loss spikes
Causa 1: Bad batch (datos corruptos)
  Síntoma: loss sube bruscamente y luego vuelve a la normalidad
  Diagnóstico: el spike dura 1-5 steps
  Solución: revisar el batch en el step que generó el spike;
            agregar filtros de calidad más agresivos al dataset

Causa 2: Learning rate muy alto
  Síntoma: spikes frecuentes o loss que no converge
  Diagnóstico: spikes al inicio del training o tras cambio de LR
  Solución: reducir LR a 1/3 o usar un warmup más largo

Causa 3: Overflow numérico en bf16
  Síntoma: gradients nan/inf reportados en debug/nan_grad
  Diagnóstico: aparece junto con grad_norm -> inf
  Solución: reducir max_grad_norm, verificar que usas bf16 (no fp16)
            bf16 tiene mayor rango que fp16 para LLMs

Causa 4: Gradient clipping insuficiente
  Síntoma: grad_norm sube a 50-100x lo normal justo antes del spike
  Solución: reducir max_norm de 1.0 a 0.5 o incluso 0.3
python — recovery tras loss spike
SPIKE_THRESHOLD = 0.5   # subida de loss que consideramos spike
prev_loss = None

for step, batch in enumerate(train_dataloader):
    with accelerator.accumulate(model):
        outputs = model(**batch)
        loss = outputs.loss
        accelerator.backward(loss)

        if accelerator.sync_gradients:
            grad_norm = accelerator.clip_grad_norm_(model.parameters(), max_norm=1.0)

            # Detectar spike y saltar el step
            if prev_loss is not None and (loss.item() - prev_loss) > SPIKE_THRESHOLD:
                print(f"⚠️  Loss spike en step {global_step}: {prev_loss:.3f} → {loss.item():.3f}")
                optimizer.zero_grad()  # descartar gradientes malos
                continue              # no actualizar pesos

        optimizer.step()
        scheduler.step()
        optimizer.zero_grad()
        prev_loss = loss.item()

3. NaN e Inf en gradientes

python — detectar y localizar NaN
import torch


# Activar detección automática de NaN/Inf (lento, solo para debugging)
torch.autograd.set_detect_anomaly(True)
# IMPORTANTE: desactivar después de debuggear, impacto severo en performance


def find_nan_parameters(model) -> list[str]:
    """Encuentra capas con NaN en sus pesos o gradientes."""
    bad_params = []
    for name, param in model.named_parameters():
        if param.data is not None and torch.isnan(param.data).any():
            bad_params.append(f"WEIGHT_NaN: {name}")
        if param.grad is not None and torch.isnan(param.grad).any():
            bad_params.append(f"GRAD_NaN: {name}")
    return bad_params


def check_batch_for_nan(batch: dict) -> bool:
    """Detecta NaN en el batch de entrada."""
    for key, tensor in batch.items():
        if isinstance(tensor, torch.Tensor) and torch.isnan(tensor).any():
            print(f"NaN en batch[{key}]")
            return True
    return False


# Causas comunes:
# 1. Datos con caracteres especiales → ID de token inválidos
# 2. fp16 (no bf16) con valores muy grandes → overflow
# 3. LayerNorm / embedding con varianzas explosivas al inicio
# Solución general: verificar datos, usar bf16 en vez de fp16

4. OOM errors (Out of Memory)

text — leer el stack trace de CUDA OOM
torch.cuda.OutOfMemoryError: CUDA out of memory.
Tried to allocate 2.50 GiB (GPU 0; 79.20 GiB total capacity;
67.35 GiB already allocated; 1.98 GiB free; 73.18 GiB reserved in total)

Cómo leer esto:
  "already allocated" = lo que el modelo + optimizador + KV cache usan
  "reserved" = lo que PyTorch reservó (puede ser mayor que "allocated")
  "free" = VRAM disponible para nueva asignación

Estrategias de reducción de memoria (en orden de impacto):
  1. Reducir batch_size (más gradient_accumulation_steps para compensar)
  2. Habilitar gradient checkpointing:
       model.gradient_checkpointing_enable()
     → intercambia memoria por recompute en backward pass
  3. Usar DeepSpeed ZeRO-2 o ZeRO-3 (distribuye optimizer states)
  4. Usar bf16 (si no lo estás usando)
  5. Reducir max_seq_len
  6. Usar LoRA / QLoRA para fine-tuning
python — gradient checkpointing + memory tracking
import torch

# Habilitar gradient checkpointing (recomendado para modelos grandes)
model.gradient_checkpointing_enable()

# Ver uso de memoria GPU antes y después de cargar el modelo
def print_gpu_memory(label: str = ""):
    allocated = torch.cuda.memory_allocated() / 1e9
    reserved = torch.cuda.memory_reserved() / 1e9
    max_alloc = torch.cuda.max_memory_allocated() / 1e9
    print(f"[{label}] Allocated: {allocated:.1f}GB | Reserved: {reserved:.1f}GB | Peak: {max_alloc:.1f}GB")

print_gpu_memory("before model load")
model = load_model(...)
print_gpu_memory("after model load")

# Limpiar caché entre evaluaciones
torch.cuda.empty_cache()

5. Memory profiling con torch.profiler

python — profiling de un step de training
import torch
from torch.profiler import profile, record_function, ProfilerActivity


def profile_training_step(model, batch, optimizer):
    """Perfila un step para identificar cuellos de botella de memoria."""
    with profile(
        activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
        profile_memory=True,
        record_shapes=True,
        with_stack=False,
    ) as prof:
        with record_function("forward"):
            outputs = model(**batch)
            loss = outputs.loss

        with record_function("backward"):
            loss.backward()

        with record_function("optimizer_step"):
            optimizer.step()
            optimizer.zero_grad()

    # Mostrar las operaciones que más memoria consumen
    print(prof.key_averages().table(
        sort_by="self_cuda_memory_usage",
        row_limit=15,
    ))

    # Guardar para visualizar en TensorBoard o Chrome Tracing
    prof.export_chrome_trace("profiling_trace.json")
Interpretar el profiler: Busca operaciones con self_cuda_memory_usage alto que no sean el modelo ni el optimizer. Los activations grandes en capas de atención son normales; tensores temporales inusuales pueden indicar operaciones ineficientes o memory leaks.

6. Checklist antes de lanzar un run largo

text — pre-launch checklist
□ Run de 100 steps en datos reales (no toy data) sin errores
□ Loss curve baja en los primeros 50 steps (confirma que el modelo aprende)
□ Gradient norm estable (< 2.0) durante warmup
□ No hay NaN/Inf en debug/nan_grad ni debug/inf_grad
□ GPU utilization > 90% (revisar con nvidia-smi)
□ DataLoader no es el cuello de botella (GPU no espera datos)
□ Checkpoint de prueba guardado y cargado correctamente
□ W&B run inicializado y recibiendo métricas
□ Script SLURM con logs activados y notificación por email
□ Suficiente espacio en disco para checkpoints (estimar: 4 x tamaño_modelo)
