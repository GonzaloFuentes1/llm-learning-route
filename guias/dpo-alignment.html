Alineamiento con DPO y variantes

Cómo alinear un modelo SFT con preferencias humanas sin entrenar un reward model explícito. Cubre DPO clásico, ORPO (sin reference model) y KTO (sin pares), con código completo usando la librería TRL.

1. RLHF vs DPO: la diferencia clave

RLHF entrena un reward model separado y luego usa PPO — costoso en VRAM y complejo de estabilizar. DPO reformula el mismo objetivo como clasificación binaria directamente sobre el modelo de lenguaje.

MétodoDatos requeridosReference modelCuándo usarlo
DPOPares (chosen, rejected)Sí (frozen)Caso estándar; tienes pares de comparación
ORPOPares (chosen, rejected)NoAhorras VRAM; menos hiperparámetros
KTOEjemplos con label bueno/maloSí (frozen)No tienes pares; solo feedback binario
SimPOPares (chosen, rejected)NoORPO con normalización por longitud

3. Entrenamiento DPO con TRL

python — dpo_train.py
import torch
from datasets import load_dataset
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig
from trl import DPOTrainer, DPOConfig

SFT_MODEL = "models/sft-checkpoint"   # modelo SFT ya entrenado
OUTPUT_DIR = "models/dpo-aligned"

tokenizer = AutoTokenizer.from_pretrained(SFT_MODEL)
tokenizer.pad_token = tokenizer.eos_token

model = AutoModelForCausalLM.from_pretrained(
    SFT_MODEL,
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

dataset = load_dataset("HuggingFaceH4/ultrafeedback_binarized", split="train_prefs")
dataset = dataset.filter(lambda x: x["score_chosen"] - x["score_rejected"] >= 1.0)

lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
)

training_args = DPOConfig(
    output_dir=OUTPUT_DIR,
    num_train_epochs=1,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=5e-7,          # DPO requiere LR mucho más bajo que SFT
    beta=0.1,                    # temperatura KL: 0.01 conservador, 0.5 agresivo
    bf16=True,
    logging_steps=10,
    save_steps=200,
    lr_scheduler_type="cosine",
    warmup_ratio=0.1,
    report_to="wandb",
    max_length=2048,
    max_prompt_length=512,
)

trainer = DPOTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
    tokenizer=tokenizer,
    peft_config=lora_config,
)

trainer.train()
trainer.save_model(OUTPUT_DIR)
Beta (β): controla cuánto se puede alejar el modelo del SFT de referencia. β bajo (0.01) = DPO muy agresivo, puede olvidar capacidades. β alto (0.5) = cambio mínimo. Empieza con β=0.1 y ajusta según las métricas.

4. ORPO: sin reference model

ORPO combina SFT y alignment en un solo paso, eliminando la necesidad de un modelo de referencia frozen. Ideal cuando tienes poco VRAM.

python — orpo_train.py
from trl import ORPOTrainer, ORPOConfig

# Misma configuración que DPO pero sin reference model
training_args = ORPOConfig(
    output_dir="models/orpo-aligned",
    num_train_epochs=1,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=8e-6,          # ORPO usa LR más alto que DPO (incluye SFT)
    beta=0.1,                    # peso del término de odds ratio
    bf16=True,
    max_length=2048,
    max_prompt_length=512,
    logging_steps=10,
    report_to="wandb",
)

trainer = ORPOTrainer(
    model=model,           # no necesita reference_model
    args=training_args,
    train_dataset=dataset,
    tokenizer=tokenizer,
    peft_config=lora_config,
)
trainer.train()

5. Métricas de diagnóstico en W&B

TRL loggea automáticamente estas métricas. Interpretarlas correctamente es clave para saber si el alineamiento está funcionando.

MétricaQué mideSeñal saludable
rewards/chosenLog-prob relativo al ref model en chosenSube durante entrenamiento
rewards/rejectedLog-prob relativo al ref model en rejectedBaja durante entrenamiento
rewards/marginschosen - rejectedPositivo y creciente
logps/chosenLog-prob absoluta en chosenNo debe caer demasiado (olvido)
lossDPO loss totalDecrece, sin spikes
Señal de alarma: si logps/chosen cae mucho (modelo olvidando el SFT) sube el β. Si rewards/margins no sube en las primeras 200 iteraciones, los datos de preferencia tienen poco contraste — filtra pares con score_chosen - score_rejected < 1.
