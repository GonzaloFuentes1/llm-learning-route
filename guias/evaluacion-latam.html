Evaluación para español/portugués con lm-eval-harness

Cómo correr benchmarks multilingüe en la práctica: instalación de lm-evaluation-harness, ejecución de MGSM, FLORES-200 e IFEval, creación de tasks personalizados para español/portugués y detección de contaminación.

1. Instalación de lm-evaluation-harness

bash
git clone https://github.com/EleutherAI/lm-evaluation-harness
cd lm-evaluation-harness
pip install -e ".[vllm]"    # instala soporte para vLLM (recomendado para velocidad)
# pip install -e ".[api]"   # si vas a evaluar via API (OpenAI-compatible)
Backend recomendado: usa --model vllm para modelos locales (3-5x más rápido que HuggingFace estándar). Requiere GPU con suficiente VRAM para el modelo.

2. Correr benchmarks estándar

Comando base

bash
# Benchmarks de conocimiento general
lm_eval \
  --model vllm \
  --model_args pretrained=models/mi-modelo-es,dtype=bfloat16,max_model_len=4096 \
  --tasks mmlu,arc_challenge,hellaswag \
  --num_fewshot 5 \
  --batch_size auto \
  --output_path resultados/mmlu_run1.json \
  --log_samples

# IFEval (instruction following — siempre 0-shot)
lm_eval \
  --model vllm \
  --model_args pretrained=models/mi-modelo-es \
  --tasks ifeval \
  --num_fewshot 0 \
  --batch_size auto \
  --output_path resultados/ifeval.json

Benchmarks multilingüe clave

bash
# MGSM — matemáticas en español (3-shot, chain-of-thought)

FLORES-200: es un benchmark de traducción, no de generación. Para evaluarlo correctamente necesitas un script dedicado con sacrebleu o chrF. Ver sección 4 de esta guía.

3. Suite de evaluación Latam (script SLURM)

Corre todos los benchmarks relevantes de un tirón en el cluster. Útil para comparar checkpoints.

bash — eval_latam.sh
#!/bin/bash
#SBATCH --job-name=eval-latam
#SBATCH --gres=gpu:1
#SBATCH --cpus-per-task=8
#SBATCH --mem=40G
#SBATCH --time=04:00:00
#SBATCH --output=logs/eval-%j.out

MODEL_PATH=$1   # pasar como argumento: sbatch eval_latam.sh models/checkpoint-1000
OUTPUT_DIR="resultados/$(basename $MODEL_PATH)"
mkdir -p $OUTPUT_DIR

TASKS="mmlu,arc_challenge,hellaswag,mgsm_es,global_mmlu_es,ifeval"

lm_eval \
  --model vllm \
  --model_args pretrained=$MODEL_PATH,dtype=bfloat16,max_model_len=4096 \
  --tasks $TASKS \
  --num_fewshot 5 \
  --batch_size auto \
  --output_path $OUTPUT_DIR/results.json \
  --log_samples

echo "Evaluación terminada: $OUTPUT_DIR/results.json"
bash — array job para múltiples checkpoints
#!/bin/bash
#SBATCH --array=0-4        # 5 checkpoints
#SBATCH --gres=gpu:1
#SBATCH --time=04:00:00

  "models/checkpoint-500"

bash eval_latam.sh ${CHECKPOINTS[$SLURM_ARRAY_TASK_ID]}

4. Crear un task personalizado

Para benchmarks propios (ej. Trueque, LatamQA) se crea un YAML de configuración. No hace falta tocar código Python.

yaml — tasks/mi_benchmark_es/mi_benchmark_es.yaml
task: mi_benchmark_es
dataset_path: huggingface_user/mi-benchmark   # dataset en HF Hub
dataset_name: es
test_split: test
output_type: multiple_choice
doc_to_text: "Pregunta: {{question}}\nOpciones:\nA. {{choice_a}}\nB. {{choice_b}}\nC. {{choice_c}}\nD. {{choice_d}}\nRespuesta:"
doc_to_target: "{{answer}}"
doc_to_choice: ["A", "B", "C", "D"]
metric_list:
  - metric: acc
    aggregation: mean
    higher_is_better: true
  - metric: acc_norm
    aggregation: mean
    higher_is_better: true
num_fewshot: 5
fewshot_config:
  sampler: first_n
bash — correr el task custom
lm_eval \
  --model vllm \
  --model_args pretrained=models/mi-modelo-es \
  --tasks mi_benchmark_es \
  --include_path tasks/mi_benchmark_es \
  --output_path resultados/mi_benchmark.json

5. Interpretar resultados correctamente

Estructura del JSON de salida

python — leer resultados
import json

with open("resultados/mmlu_run1.json") as f:
    results = json.load(f)

# Accuracy por task
for task, metrics in results["results"].items():
    acc = metrics.get("acc,none", metrics.get("acc_norm,none"))
    stderr = metrics.get("acc_stderr,none", metrics.get("acc_norm_stderr,none", 0))
    print(f"{task}: {acc:.3f} ± {stderr:.3f}")

# Comparar dos modelos
import pandas as pd

def load_results(path):
    with open(path) as f:
        data = json.load(f)
    return {task: m.get("acc,none", m.get("acc_norm,none"))
            for task, m in data["results"].items()}

base   = load_results("resultados/baseline/results.json")
tuned  = load_results("resultados/finetuned/results.json")
df = pd.DataFrame({"baseline": base, "finetuned": tuned})
df["delta"] = df["finetuned"] - df["baseline"]
print(df.sort_values("delta", ascending=False))
Errores comunes de reporting (ver arXiv:2511.21140):
  • Reportar solo los mejores resultados entre varias ejecuciones (cherry-picking)
  • No reportar el número de few-shot shots usados
  • Comparar modelos evaluados con prompts diferentes
  • Usar acc vs acc_norm inconsistentemente — acc_norm penaliza por longitud de opción

6. Detección de contaminación

La contaminación ocurre cuando los datos de evaluación aparecen en el training set. Puede inflar scores 5–15 puntos. Detectarla con n-gram overlap:

python — contamination_check.py
from datasets import load_dataset
from collections import Counter
import re

def ngrams(text, n=13):
    tokens = text.lower().split()
    return set(zip(*[tokens[i:] for i in range(n)]))

# Cargar benchmark y subset del training data
benchmark = load_dataset("cais/mmlu", "all", split="test")
train_sample = load_dataset("HuggingFaceFW/fineweb", split="train", streaming=True)

# Construir set de n-grams del training (muestra de 1M docs)
train_ngrams = set()
for i, doc in enumerate(train_sample):
    if i >= 1_000_000:
        break
    train_ngrams.update(ngrams(doc["text"]))

# Verificar contaminación en el benchmark
contaminated = 0
for item in benchmark:
    q_ngrams = ngrams(item["question"])
    if q_ngrams & train_ngrams:  # intersección no vacía
        contaminated += 1

print(f"Posible contaminación: {contaminated}/{len(benchmark)} ({100*contaminated/len(benchmark):.1f}%)")
