Uso del Cluster HPC

Guía de referencia para conectarse y operar el cluster de cómputo del proyecto: llaves SSH, VS Code remoto, transferencia de archivos, monitoreo de recursos y gestión de jobs SLURM.

1. Llaves SSH

SSH usa criptografía de clave pública: tú tienes una clave privada (secreta, en tu máquina) y el servidor guarda tu clave pública. Nunca compartas la clave privada.

Generar un par de claves
bash
# Genera un par de claves Ed25519 (más moderno y seguro que RSA)
ssh-keygen -t ed25519 -C "tu@email.com"

# Acepta la ubicación por defecto: ~/.ssh/id_ed25519
# Puedes agregar una passphrase (recomendado) o dejarla vacía

# Ver las claves generadas
ls ~/.ssh/
# id_ed25519       ← clave privada (NUNCA compartas esto)
# id_ed25519.pub   ← clave pública (ésta la compartes)
Compartir tu clave pública con el servidor
bash
# Opción 1: usar ssh-copy-id (la forma más simple)
ssh-copy-id -i ~/.ssh/id_ed25519.pub usuario@cluster.host

# Opción 2: copiar manualmente (si no tienes ssh-copy-id)
cat ~/.ssh/id_ed25519.pub
# Copia el output y pégalo en ~/.ssh/authorized_keys en el servidor

# En el servidor, el archivo queda así:
# ~/.ssh/authorized_keys
# ssh-ed25519 AAAA...clave... tu@email.com
Permisos importantes: El directorio ~/.ssh debe tener permisos 700 y authorized_keys debe tener 600. SSH rechaza conexiones si los permisos son demasiado abiertos: chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys

2. Conexión por terminal

Conexión básica
bash
# Conexión directa
ssh usuario@cluster.host

# Con puerto no estándar
ssh -p 2222 usuario@cluster.host

# Con clave específica
ssh -i ~/.ssh/id_ed25519 usuario@cluster.host
Configurar ~/.ssh/config para aliases

En lugar de recordar el host y usuario, crea un alias en ~/.ssh/config:

ssh config
# ~/.ssh/config

# Cluster principal
Host latamgpt
    HostName cluster.cenia.cl
    User tu_usuario
    IdentityFile ~/.ssh/id_ed25519
    ServerAliveInterval 60

# Con bastion (salto intermedio)
Host latamgpt
    HostName 10.0.0.50
    User tu_usuario
    ProxyJump bastion.cenia.cl
    IdentityFile ~/.ssh/id_ed25519
bash
# Ahora puedes conectarte simplemente con:
ssh latamgpt

3. VS Code Remote-SSH

Puedes editar código en el cluster directamente desde VS Code en tu máquina local, con autocompletado, git y terminal integrados.

Setup
  1. Instala la extensión Remote - SSH (Microsoft) en VS Code
  2. Abre la paleta de comandos: Ctrl+Shift+P → Remote-SSH: Connect to Host
  3. Selecciona el alias de tu ~/.ssh/config (ej: latamgpt)
  4. VS Code se conecta, instala el servidor remoto y abre una nueva ventana
  5. Abre carpeta: File → Open Folder → navega a tu directorio de trabajo
Tip: Una vez conectado, la terminal integrada (Ctrl+`) abre un shell directamente en el cluster. El explorador de archivos, git y extensiones de Python funcionan sobre el servidor remoto.

4. Transferir archivos

rsync (recomendado para directorios)
bash
# Subir un directorio al cluster (con progreso)
rsync -avz --progress ./mi_proyecto/ latamgpt:~/proyectos/mi_proyecto/

# Bajar resultados desde el cluster
rsync -avz --progress latamgpt:~/outputs/run_001/ ./outputs/

# Opciones útiles:
#  -a  preserve permisos, timestamps, symlinks
#  -v  verbose (muestra qué se copia)
#  -z  comprimir en tránsito
#  --exclude='*.pyc' --exclude='__pycache__'  ignorar archivos
#  --dry-run  simular sin copiar nada
scp (archivos individuales)
bash
# Subir un archivo
scp ./config.yaml latamgpt:~/proyectos/

# Bajar un archivo
scp latamgpt:~/outputs/model.pt ./

# Con puerto no estándar
scp -P 2222 archivo.txt latamgpt:~/

5. Monitoreo CPU y RAM

htop (interactivo)
bash
htop

# Atajos dentro de htop:
# F5  → vista de árbol de procesos
# F6  → ordenar por columna (CPU%, MEM%)
# /   → filtrar por nombre de proceso
# k   → enviar señal (matar proceso)
# q   → salir

# Para ver solo tus procesos:
htop -u $USER
btop (más visual, alternativa moderna)
bash
btop
# Muestra CPU por core, RAM, disco, red y procesos
# Ideal para diagnóstico rápido de cuellos de botella
Qué mirar
bash
# Revisar memoria disponible rápido
free -h
# Salida: total used free shared buff/cache available

# Load average (carga del sistema)
uptime
# "load average: 32.5, 30.1, 28.4" → últimos 1, 5 y 15 minutos
# Un load average > número de CPUs indica cuello de botella

6. Monitoreo GPU

nvidia-smi (GPUs NVIDIA)
bash
# Vista estática
nvidia-smi

# Vista en tiempo real (refresca cada 1 segundo)
watch -n 1 nvidia-smi

# Columnas importantes:
# GPU%       → utilización del compute (debería ser ~95-100% durante training)
# MEM-Usage  → VRAM usada / total (ej: 78341MiB / 80000MiB)
# Temp       → temperatura en °C (alerta si > 85°C)

# Ver solo GPU 0 y 1
nvidia-smi -i 0,1

# Formato de monitor continuo (parseable)
nvidia-smi dmon -s pucvmet -d 2
nvtop (interactivo, como htop para GPUs)
bash
nvtop
# Muestra todas las GPUs, procesos por GPU, memoria y utilización en tiempo real
# Usa flechas para navegar entre GPUs
rocm-smi (GPUs AMD)
bash
rocm-smi
# Equivalente a nvidia-smi para tarjetas AMD (MI250, MI300, etc.)

watch -n 1 rocm-smi
GPU utilization vs MEM utilization: Durante training el GPU% debería estar cerca del 100%. Si está bajo, el cuello de botella puede ser la carga de datos (DataLoader lento) o que el batch size es muy pequeño. nvtop te muestra qué proceso usa qué GPU.

7. Gestionar GPUs con CUDA

La variable de entorno CUDA_VISIBLE_DEVICES controla qué GPUs son visibles para tu proceso. Es esencial para correr múltiples trabajos en el mismo nodo.

bash
# Usar solo la GPU 0
CUDA_VISIBLE_DEVICES=0 python train.py

# Usar las GPUs 0 y 1
CUDA_VISIBLE_DEVICES=0,1 python train.py

# Usar las GPUs 2, 3, 4 y 5 (dejando 0 y 1 para otros)
CUDA_VISIBLE_DEVICES=2,3,4,5 python train.py

# Deshabilitar todas las GPUs (forzar CPU)
CUDA_VISIBLE_DEVICES="" python script.py

# Verificar qué GPUs ve tu script desde Python
python -c "import torch; print(torch.cuda.device_count())"
Importante: Los índices dentro de CUDA_VISIBLE_DEVICES son re-mapeados. Si usas CUDA_VISIBLE_DEVICES=2,3, dentro de tu código esas GPUs serán cuda:0 y cuda:1 respectivamente.

8. Procesos en background con tmux

El training puede durar horas o días. tmux mantiene procesos vivos aunque cierres la terminal SSH.

bash
# Crear nueva sesión llamada "train"
tmux new -s train

# Dentro de la sesión: lanzar tu proceso
python train.py

# Desconectarse (el proceso sigue corriendo)
Ctrl+B  d     # → d de "detach"

# Ver sesiones activas
tmux ls

# Reconectarse a la sesión "train"
tmux attach -t train

# Crear múltiples ventanas dentro de la misma sesión
Ctrl+B  c     # nueva ventana
Ctrl+B  ,     # renombrar ventana
Ctrl+B  n     # siguiente ventana
Ctrl+B  p     # ventana anterior
Ctrl+B  0-9   # ir a ventana por número

# Dividir ventana en paneles
Ctrl+B  %     # split vertical
Ctrl+B  "     # split horizontal
Ctrl+B  flechas  # moverse entre paneles

# Matar sesión
tmux kill-session -t train
Alternativa: screen es más simple pero menos poderoso: screen -S train para crear, Ctrl+A d para detach, screen -r train para reconectar.

9. SLURM: gestor de jobs

SLURM es el scheduler del cluster. En lugar de correr procesos directamente, envías "jobs" que SLURM programa según recursos disponibles.

Comandos básicos
bash
# Enviar un job
sbatch mi_job.sh

# Ver jobs en cola (propios y de todos)
squeue -u $USER        # solo los tuyos
squeue                 # todos

# Cancelar un job
scancel 12345          # por job ID
scancel -u $USER       # todos los tuyos

# Ver información detallada de un job
scontrol show job 12345

# Ver particiones disponibles y sus GPUs
sinfo
Template de script SLURM
bash — mi_job.sh
#!/bin/bash
#SBATCH --job-name=latamgpt-train     # nombre del job
#SBATCH --partition=gpu               # partición con GPUs
#SBATCH --gres=gpu:4                  # número de GPUs
#SBATCH --ntasks=1                    # número de tareas
#SBATCH --cpus-per-task=32            # CPUs por tarea
#SBATCH --mem=256G                    # RAM total
#SBATCH --time=24:00:00               # tiempo límite hh:mm:ss
#SBATCH --output=logs/train_%j.out    # stdout (%j = job ID)
#SBATCH --error=logs/train_%j.err     # stderr
#SBATCH --mail-type=END,FAIL          # notificación por email
#SBATCH --mail-user=tu@email.com

# Activar entorno
source ~/.bashrc
conda activate latamgpt

# Variables
export CUDA_VISIBLE_DEVICES=0,1,2,3
export OMP_NUM_THREADS=8

# Lanzar el training
accelerate launch \
  --config_file configs/deepspeed_zero2.yaml \
  --num_processes 4 \
  train.py \
  --config configs/pretraining.yaml

echo "Job terminado: $(date)"
bash
# Seguir los logs en tiempo real
tail -f logs/train_12345.out

# Ver el uso de recursos de un job en ejecución
sstat --format=JobID,AveCPU,AveRSS,MaxRSS -j 12345

10. Git básico en el cluster

bash
# Clonar el repositorio del proyecto
git clone git@github.com:latamgpt/llm-training.git
cd llm-training

# Crear una rama para tu experimento
git checkout -b exp/tokenizer-v2

# Ver cambios antes de commitear
git status
git diff

# Commitear de forma interactiva (seleccionar qué incluir)
git add -p
git commit -m "feat: actualizar vocabulario tokenizador para es/pt"

# Subir la rama
git push origin exp/tokenizer-v2

# Traer cambios de main sin perder tu trabajo
git fetch origin
git rebase origin/main

# Ver últimos commits del proyecto
git log --oneline -20
Nunca hagas git add . en el cluster — puedes incluir accidentalmente checkpoints de cientos de GB. Usa git add -p o agrega un .gitignore apropiado con *.pt, *.bin, checkpoints/.
